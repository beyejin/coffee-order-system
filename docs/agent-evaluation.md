# 에이전트 작업 평가

AI 준비도 개선 후 대표 작업의 현재 pass-rate를 기록합니다. 점수보다 재현 가능한 과제, 통과 기준, 실제 근거를 남기는 것이 목적입니다.

## 2026-07-13 측정

### 방법

- 각 작업은 저장소 경로와 과제만 받은 독립된 초기 문맥의 에이전트가 수행했습니다.
- 모든 에이전트는 루트 `AGENTS.md`부터 읽었고 저장소 파일은 수정하지 않았습니다.
- 최초 과제 전달 이후 추가 힌트나 교정 프롬프트가 필요하면 사용자 개입 1회로 계산했습니다.
- 기능 변경 과제는 불필요한 제품 변경을 만들지 않도록 구현 직전 계획과 검증 설계까지만 평가했습니다.

### 결과

| 유형 | 과제 | 통과 기준 | 결과 | 추가 개입 |
|---|---|---|---|---:|
| 탐색 | `POST /orders`에서 포인트·주문·이벤트·외부 전송·랭킹까지 추적 | 실제 파일과 메서드, 트랜잭션 경계, 외부 전송 실패의 롤백 여부, 랭킹 집계 방식을 모두 식별 | **PASS** — `OrderService.order()`의 DB 원자성, `AFTER_COMMIT` 전송, 주문 원장 직접 집계를 정확히 식별 | 0 |
| 기능 변경 | 충전 후 잔액 1,000,000P 상한 도입 계획 | 기존 정책 충돌, 문서 우선 변경, 최소 코드 경로, 경계값·롤백·동시 요청 MySQL 테스트를 식별 | **PASS** — `strategy/policy/api-spec` 충돌과 `User`, `ErrorCode`, `PointChargeIntegrationTest`의 최소 변경을 식별 | 0 |
| 회귀 검증 | 문서·컨텍스트 검사, migration 단독 테스트, 전체 테스트 실행 | 명령을 실제 실행하고 성공 여부와 테스트 개수를 보고 | **PASS** — 정적 검사 PASS, migration 1/1, 전체 34/34 PASS | 0 |

**현재 pass-rate: 3/3 (100%)**

**총 추가 개입: 0회**

기능 변경 에이전트는 구현을 시작하려면 기존 미커밋 변경의 기준선 확정과 GitHub 이슈가 필요하다고 별도로 지적했습니다. 이는 `AGENTS.md`의 구현 게이트를 따른 중단 판단이며, 평가 도중 추가 힌트를 요구한 것은 아니므로 개입 횟수에는 포함하지 않았습니다.

### 실행 근거

- 문서·컨텍스트: `python3 scripts/check-doc-context.py` → Markdown 링크 PASS, `src` 커버리지 100%
- migration: `./gradlew test --tests com.example.coffee.MigrationUpgradeTest --rerun-tasks --console plain` → 1개 통과
- 전체 회귀: `./gradlew clean test --console plain` → 34개 통과, 실패·오류·건너뜀 0

### 해석 한계

- 각 유형을 한 번씩 실행한 작은 표본이므로 100%를 일반적인 성공률로 해석할 수 없습니다.
- 기능 변경은 계획까지만 평가했으며 실제 코드 작성 품질은 측정하지 않았습니다.
- 개선 전 같은 과제를 실행한 기준선이 없어 전후 향상 폭은 계산하지 않았습니다.
- 다음 측정에서는 동일 과제를 여러 초기 문맥에서 반복하고, 구현 과제는 격리된 브랜치에서 테스트 통과까지 평가합니다.
