# 개발 방법 (Conventions)

## 문서 작성

- 설계를 바꾸는 코드 변경 전에는 관련 `docs/*.md`(진실의 원천)를 먼저 갱신한다. 문서 주도 개발(document-driven development)이 원칙이다 — 코드가 문서를 따라가야지, 문서가 코드를 사후 설명해서는 안 된다.
- 모든 문서는 한국어로 작성한다. 코드 식별자·기술 용어는 영어를 유지해도 된다.
- `docs/strategy.md` 5장의 "검토 중인 전략" 표는 결정이 바뀔 때마다 갱신한다 — 오래된 트레이드오프 표를 방치하지 않는다.

## 커밋 메시지

Conventional Commits 형식(`type: 설명`)을 한국어로 작성한다.

| type | 용도 |
|---|---|
| `feat` | 기능 추가 |
| `fix` | 버그 수정 |
| `test` | 테스트 추가/수정 |
| `docs` | 문서 변경 |
| `refactor` | 동작 변화 없는 구조 개선 |
| `chore` | 빌드/설정 등 그 외 |

예: `feat: 포인트 충전 API 구현`, `test: 동시 주문 시나리오 테스트 추가`, `docs: ERD에 인덱스 설계 추가`

## 로그 / 작업 기록

- 별도의 작업 로그 파일을 두지 않는다. **커밋 히스토리가 작업 로그를 대신한다** — 커밋 메시지 본문에 "무엇을 왜 바꿨는지"를 남긴다.
- 설계 판단의 근거(트레이드오프, 왜 이 방식을 선택했는지)는 커밋이 아니라 `docs/strategy.md`에 남긴다 — 커밋 로그는 휘발성이 있고 문서가 진실의 원천이어야 하기 때문이다.
