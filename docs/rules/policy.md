# 정책 (Policy) — 공유 비즈니스 규칙 / 진실의 원천

여러 기능이 함께 의존하는 도메인 불변식과, 아직 결정되지 않은 사항을 어떻게 다룰지에 대한 규칙이다. 명세(테이블 구조는 `docs/erd.md`, API 형태는 `docs/api-spec.md`)와는 다르게, "여러 기능에 걸쳐 항상 참이어야 하는 규칙"만 다룬다.

## 도메인 불변식 — 절대 깨지 않는다

- `Order`에 `status` 컬럼을 추가하지 않는다 — 실패한 주문은 레코드 자체를 만들지 않는 것이 설계 전제다 (`존재하는 주문 = 성공한 주문`).
- `User.balance`와 `PointHistory` INSERT는 항상 같은 트랜잭션에서 함께 갱신한다.
- `Order.price`는 주문 시점 스냅샷이다 — `Menu` 조인으로 대체하지 않는다.
- 주문은 요청 1건당 메뉴 1개다 — 장바구니(여러 메뉴) 구조로 확장하지 않는다.

이 규칙들의 배경과 근거는 `docs/strategy.md` 2장 참고.

## 보류 항목 — 임의로 최종 결정하지 않는다

`docs/strategy.md` 5장의 "검토 중인 전략" 3가지는 사용자가 아직 학습 중인 주제다:

1. 포인트 차감 동시성 제어
2. 결제 트랜잭션 ↔ 데이터 수집 플랫폼 실시간 전송
3. 인기 메뉴(최근 7일) 집계 방식

각 항목의 **현재 기본안**(비관적 락 / 트랜잭션 내 동기 Mock / 실시간 GROUP BY)까지만 구현한다. 더 나은 대안(Redis 분산 락, Outbox 패턴, 캐싱 등)으로 바꾸고 싶으면 먼저 사용자에게 제안하고 승인받은 뒤 `strategy.md`를 갱신하고 구현한다 — 코드에서 먼저 바꾸고 문서로 사후 설명하지 않는다.

## 검증 정책

테스트는 Testcontainers로 실제 MySQL 컨테이너를 띄워 검증한다. H2 등 인메모리 DB로 대체하지 않는다 — 락/트랜잭션 동작이 달라 동시성 관련 검증을 신뢰할 수 없기 때문이다.
