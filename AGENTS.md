# AGENTS.md

이 저장소에서 작업하는 AI 에이전트(Claude Code, Codex 등)가 따라야 할 규칙입니다.

## 프로젝트 컨텍스트

K사 서버 개발 채용 사전과제 — 커피숍 주문 시스템. 정답이 정해진 과제가 아니라 "왜 이렇게 설계했는가"를 설명하는 것이 평가 핵심입니다.

**항상 먼저 읽을 것**: [`docs/strategy.md`](docs/strategy.md), [`docs/erd.md`](docs/erd.md), [`docs/api-spec.md`](docs/api-spec.md) — 이 세 문서가 설계의 단일 진실 원천(source of truth)입니다. 코드가 문서와 다르게 동작하면 버그입니다.

## 절대 규칙

1. **도메인 불변식을 깨지 마라.**
   - `Order`에 `status` 컬럼을 추가하지 않는다 — 실패한 주문은 레코드 자체를 만들지 않는 것이 설계 전제다.
   - `User.balance`와 `PointHistory` INSERT는 항상 같은 트랜잭션에서 함께 갱신한다.
   - `Order.price`는 주문 시점 스냅샷이다 — `Menu` 조인으로 대체하지 않는다.
   - 주문은 요청 1건당 메뉴 1개다 — 장바구니(여러 메뉴) 구조로 확장하지 않는다.

2. **`docs/strategy.md` 5장의 "검토 중인 전략" 3가지(동시성 제어, 이벤트 전송, 인기 메뉴 집계)는 사용자가 아직 학습 중인 주제다.**
   임의로 최종 구현을 밀어붙이지 말 것. 현재 "기본안"으로 명시된 방식(비관적 락 / 트랜잭션 내 동기 Mock / 실시간 GROUP BY)까지만 구현하고, 더 나은 대안(Redis 분산 락, Outbox 패턴, 캐싱 등)으로 바꾸고 싶으면 **먼저 사용자에게 제안하고 승인받은 뒤** `strategy.md`를 갱신하고 구현한다.

3. **설계를 바꾸는 코드 변경 전에는 먼저 관련 `docs/*.md`를 갱신한다.** 문서 주도 개발(document-driven development)이 이 프로젝트의 워크플로우다 — 코드가 문서를 따라가야지, 문서가 코드를 사후 설명해서는 안 된다.

4. **테스트는 Testcontainers로 실제 MySQL 컨테이너를 띄워 검증한다.** H2 등 인메모리 DB로 대체하지 않는다 (락/트랜잭션 동작이 달라 신뢰할 수 없음).

## 워크플로우

- 구현 전 이 저장소는 튜터 설계 점검을 거친다. 점검 전에는 큰 구조 변경(엔티티 추가/삭제, API 시그니처 변경)을 임의로 하지 않는다.
- 새 기능은 해피패스 + 주요 예외 케이스(포인트 부족, 존재하지 않는 ID) 통합 테스트를 함께 작성한다.
- 커밋 메시지는 Conventional Commits 형식(`type: 설명`)을 한국어로 작성한다. 예: `feat: 포인트 충전 API 구현`, `docs: ERD에 인덱스 설계 추가`, `test: 동시 주문 시나리오 테스트 추가`.

## 커뮤니케이션

- 이 저장소에서의 모든 설명·커밋 메시지·문서는 한국어로 작성한다. 코드 식별자·기술 용어는 영어를 유지해도 된다.
- 불확실한 설계 판단(특히 위 "절대 규칙 2"의 3가지 보류 항목)은 추측해서 구현하지 말고 먼저 질문한다.
